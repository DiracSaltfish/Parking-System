# 停车场管理系统详细功能设计

## 1. 项目目标

基于你当前的实训要求，这个项目建议做成一个 **双角色前后端分离系统**：

- 前端：`Vue 3 + Element Plus`
- 后端：`Spring Boot + Java`
- 数据存储：`Redis`

系统分为两类角色：

- 管理员
- 普通用户（车主）

核心目标不是做特别复杂，而是做一个**逻辑完整、能演示、能答辩**的系统。

---

## 2. 角色与功能划分

### 2.1 管理员功能

管理员进入后台后，可以完成以下操作：

- 管理员账号登录
- 查看停车场总体运行情况
- 查看所有正在停放的车辆
- 查看所有历史停车记录
- 查看每辆车的停车时长、费用、支付状态
- 手动录入车辆入场信息
- 手动办理车辆出场
- 维护车位信息
- 设置收费规则
- 对单独车辆执行停车费豁免
- 查看今日收费统计、车位使用统计

### 2.2 普通用户功能

普通用户进入系统后，可以完成以下操作：

- 用户注册/登录
- 绑定自己的车牌号
- 查看自己当前是否在场
- 查询自己的停车时长
- 查询当前待缴费用
- 在线缴费
- 查看历史停车记录
- 查看历史缴费记录

## 4. 核心业务设计

### 4.1 管理员登录

管理员使用账号密码登录后台。

登录后返回：

- 用户基本信息
- 角色信息
- 登录 Token

说明：

- 后端统一做 Token 校验
- 前端根据角色决定进入管理员端还是用户端

### 4.2 普通用户注册与登录

普通用户建议实现最基础版本：

- 手机号/用户名注册
- 用户名密码登录

实训课如果时间紧，也可以简化为：

- 直接账号密码登录
- 不做短信验证

### 4.3 用户绑定车牌

这是普通用户功能里的关键点。

一个普通用户可以绑定：

- 1 个车牌
- 或多个车牌

为了降低实现复杂度，第一版建议：

- 一个用户绑定一个主车牌

功能包含：

- 新增绑定车牌
- 修改车牌
- 删除绑定车牌
- 查询当前绑定车牌

业务约束建议：

- 同一个车牌只能绑定一个用户
- 车牌格式要做基础校验

### 4.4 车辆入场

车辆入场由管理员操作，流程如下：

1. 管理员输入车牌号
2. 系统检查该车是否已经在场
3. 系统分配空闲车位
4. 记录入场时间
5. 更新车位状态为已占用
6. 生成停车记录

如果后续想增强，可以扩展：

- 支持选择指定车位
- 支持临时车和绑定车主车辆区分

### 4.5 用户查询当前停车信息

普通用户登录后，可以查看自己绑定车牌当前是否在停车场内。

如果车辆在场，需要展示：

- 车牌号
- 入场时间
- 已停车时长
- 当前车位
- 当前待缴费用
- 支付状态

### 4.6 用户缴费

普通用户可以对自己当前停车订单进行缴费。

实训版建议不要接真实支付平台，直接做模拟支付：

- 点击“立即缴费”
- 选择支付方式（微信/支付宝/余额，实际只做界面模拟）
- 后端更新支付状态为“已支付”
- 记录支付时间

说明：

- 这是课程项目中最常见也最稳妥的做法
- 既有完整支付业务，又不依赖第三方接口

### 4.7 管理员查看全部车辆

管理员可以在“当前停车车辆”页面查看：

- 当前所有在场车辆
- 所在车位
- 入场时间
- 已停时长
- 当前应缴金额
- 是否已缴费

可以支持的筛选条件：

- 按车牌号查询
- 按支付状态筛选
- 按是否超时筛选

### 4.8 停车费豁免

这是你特别提到的功能，可以作为一个比较亮眼的扩展点。

管理员可对某一条停车记录执行费用豁免，常见场景：

- 内部车辆免费
- 特殊情况免单
- 系统故障补偿

建议设计两种豁免方式：

- 全额豁免
- 部分减免

需要记录的字段：

- `discountType`
- `discountAmount`
- `finalAmount`
- `exemptReason`
- `operatorId`
- `operateTime`

这样后面答辩时可以讲清楚“原始费用”和“最终费用”的区别。

### 4.9 车辆出场

车辆出场可由管理员办理，也可以设计为“用户先缴费，管理员确认出场”。

建议第一版流程：

1. 管理员查询当前车辆
2. 系统计算应收费用
3. 若已缴费或已豁免，则允许出场
4. 记录出场时间
5. 释放车位
6. 将当前停车记录转为历史记录

### 4.10 收费规则

建议先做固定规则：

- 前 30 分钟免费
- 超过后每小时收费
- 不足 1 小时按 1 小时计算

例如：

- 免费时长：`30` 分钟
- 每小时：`5` 元
- 每日封顶：`30` 元

这样足够完成实训，也方便后期升级。

## 7. Redis Key 设计补充

在原有设计上，建议增加：

### 7.1 普通用户

```text
user:{userId} -> Hash
user:username:{username} -> String
```

### 7.2 用户绑定车辆

```text
user:vehicle:{vehicleId} -> Hash
user:{userId}:vehicles -> Set
vehicle:plate:{plateNumber} -> String
```

说明：

- `vehicle:plate:{plateNumber}` 可直接映射到 `userId` 或 `vehicleId`
- 用于快速判断车牌是否已绑定

### 7.3 支付记录

```text
payment:{paymentId} -> Hash
payment:user:{userId} -> List
payment:record:{recordId} -> String
```

### 7.4 费用豁免记录

```text
parking:exemption:{exemptionId} -> Hash
parking:record:exemptions:{recordId} -> List
```

## 11. 最适合答辩展示的业务流程

建议你演示这一条完整链路：

1. 普通用户注册并绑定车牌
2. 管理员录入该车辆入场
3. 普通用户登录查看当前停车时长和费用
4. 普通用户完成模拟缴费
5. 管理员查看该车辆支付状态
6. 管理员对另一辆车执行费用豁免
7. 管理员办理车辆出场
8. 查看历史停车记录和今日统计

这条流程覆盖角色切换、前后端交互、Redis 数据更新、费用计算、支付和豁免，展示效果会很好。
