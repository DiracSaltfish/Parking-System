# 停车场管理系统加密存储流程设计

## 1. 目标

这个设计用于后续把当前项目从“内存态演示”升级到“可安全存储”的实现方式。

重点保护三类数据：

- 用户密码
- 登录凭证 Token
- Redis 中的敏感业务数据

---

## 2. 总体原则

### 2.1 密码不做可逆加密，只做单向哈希

用户密码和管理员密码不能明文保存，也不建议做可逆加密。

推荐方案：

- 使用 `BCrypt` 或 `Argon2`
- 每个密码自动带盐
- 后端只保存哈希值，不保存原密码

说明：

- 登录时比较哈希，不做明文对比
- 即使 Redis 或数据库泄露，也不能直接还原密码

### 2.2 业务敏感字段做对称加密

下列字段如果需要长期存储，可以做字段级加密：

- 手机号
- 真实姓名
- 车牌号
- 支付流水号

推荐方案：

- 后端使用 `AES-GCM`
- 密钥从环境变量或配置中心读取
- 不把密钥写死在代码里

### 2.3 Token 不做长期明文持久化

推荐方式：

- Access Token 短时有效
- 前端优先放在 `sessionStorage` 或内存
- 后端 Redis 保存 Token 摘要、用户 ID、过期时间

如果后续做刷新机制：

- Refresh Token 用 `HttpOnly Cookie`

---

## 3. 推荐存储方案

## 3.1 用户表/管理员表

原始字段：

- `username`
- `password`
- `phone`
- `realName`

推荐存储：

- `username`：明文或规范化后明文
- `passwordHash`：`BCrypt`
- `phoneEncrypted`：`AES-GCM`
- `realNameEncrypted`：`AES-GCM`

说明：

- 用户名通常需要登录检索，适合明文存储
- 手机号和姓名属于敏感字段，适合加密存储

## 3.2 车牌绑定信息

如果需要按车牌精确查询，同时又要保护隐私，建议双字段：

- `plateNumberEncrypted`
- `plateNumberHash`

处理方式：

1. 原始车牌先标准化，例如统一大写
2. 用 `SHA-256` 生成查询摘要
3. 用 `AES-GCM` 保存密文

这样可以同时满足：

- 通过摘要做等值查询
- 通过密文做展示还原

## 3.3 支付信息

支付业务建议不要保存完整第三方敏感信息。

推荐保留：

- 支付方式
- 支付金额
- 支付时间
- 业务支付单号

如果有外部流水号：

- 可以只保留末尾几位展示
- 完整号做加密存储

---

## 4. 登录与存储流程

### 4.1 注册流程

1. 前端提交用户名、密码、手机号、姓名
2. 后端校验格式
3. 后端对密码执行 `BCrypt`
4. 后端对手机号、姓名执行 `AES-GCM`
5. Redis 或后续持久化存储保存加密结果

### 4.2 登录流程

1. 前端提交用户名和密码
2. 后端根据用户名取出 `passwordHash`
3. 使用 `BCrypt.matches()` 校验
4. 校验通过后生成 Token
5. Redis 保存 Token 会话信息与过期时间
6. 前端只保存 Token，不保存密码

### 4.3 用户查询资料流程

1. 后端根据当前登录用户读取加密字段
2. 后端在服务层解密
3. 返回前端显示必要信息

说明：

- 解密只发生在后端
- 前端不持有解密密钥

---

## 5. Redis 落地建议

### 5.1 账号信息

```text
user:{userId}
```

字段建议：

- `username`
- `passwordHash`
- `phoneEncrypted`
- `realNameEncrypted`

### 5.2 车牌绑定

```text
user:vehicle:{vehicleId}
```

字段建议：

- `userId`
- `plateNumberEncrypted`
- `plateNumberHash`
- `isPrimary`

### 5.3 Token 会话

```text
auth:token:{tokenId}
```

字段建议：

- `userId`
- `role`
- `issuedAt`
- `expiresAt`

说明：

- Redis Key 设置过期时间
- 不建议无限期有效

---

## 6. 密钥管理

密钥不要写在仓库里。

推荐来源：

- 操作系统环境变量
- `.env` 文件但不提交 Git
- 服务器配置中心

例如：

```text
APP_AES_KEY=your-32-byte-key
```

要求：

- AES 密钥至少 256 位
- 定期轮换
- 生产环境和开发环境使用不同密钥

---

## 7. 当前项目最适合的实现顺序

因为你现在还没有接 Redis，最合理的顺序是：

1. 先把登录密码改成 `BCrypt` 存储
2. 再把 Token 改成有过期时间的会话存储
3. 接 Redis 后，再补手机号、姓名、车牌号的 `AES-GCM` 字段加密
4. 最后补密钥管理和日志脱敏

---

## 8. 不建议的做法

- 不要明文保存密码
- 不要把 AES 密钥写死在前端
- 不要让前端负责真正的数据解密
- 不要把长期有效 Token 明文放在 `localStorage`
- 不要把敏感字段完整打印到日志

---

## 9. 对当前项目的直接建议

针对你这个停车场管理系统，后续正式实现时建议这样落地：

1. 登录密码：`BCrypt`
2. 用户手机号/真实姓名：`AES-GCM`
3. 车牌号：`Hash + AES-GCM` 双存
4. Token：短期会话，Redis 设置过期
5. 前端：只存 Token，不存密码，优先 `sessionStorage`

如果继续往下做，下一步我可以直接帮你把：

- 前端 Token 从 `localStorage` 改成 `sessionStorage`
- 后端密码存储从明文骨架改成 `BCrypt` 骨架

这两步做完，项目安全性会比现在高一层。
