# 停车场管理系统项目汇报重点

## 1. 项目背景与目标

- 针对传统停车场人工登记效率低、信息管理不便、收费不透明等问题，设计并实现一个停车场管理系统
- 项目采用前后端分离架构，完成停车管理核心业务闭环
- 系统面向两类角色：
  - 管理员
  - 普通用户
- 实现车辆入场、停车计费、费用查询、缴费、记录管理等功能

---

## 2. 技术选型

- 前端：Vue 3 + Element Plus
- 后端：Spring Boot + Java
- 数据存储：Redis
- 接口风格：RESTful API
- 开发模式：前后端分离

---

## 3. 系统架构设计

### 前端职责
- 实现页面展示
- 完成用户输入校验
- 调用后端接口获取和提交数据
- 根据用户角色跳转不同页面

### 后端职责
- 处理登录认证
- 执行业务逻辑
- 计算停车时长和停车费用
- 对接 Redis 完成数据存储
- 返回统一格式的接口数据

### 数据存储职责
- 使用 Redis 存储用户、车辆、停车记录、支付记录、Token 会话等信息
- 利用 Redis 的高读写效率满足停车场实时状态管理需求

---

## 4. 用户角色设计

### 管理员功能
- 管理员登录系统
- 查看当前所有停放车辆
- 查看历史停车记录
- 手动录入车辆入场

- 后续可扩展出场办理、车位管理、收费规则配置

### 普通用户功能
- 用户注册与登录
- 绑定个人车牌
- 查询当前停车时长
- 查询当前停车费用
- 查看历史停车记录
- 查看缴费记录
- 在线模拟缴费

## 5. 核心功能模块

- 统一登录入口
- 用户注册与登录
- 管理员登录
- 车牌绑定
- 当前停车信息查询
- 停车记录查询
- 缴费记录查询

## 6. 业务流程说明

### 普通用户流程
1. 注册账号并登录系统
2. 绑定个人车牌
3. 查询当前停车信息
4. 查看停车时长和费用
5. 完成模拟缴费
6. 查询历史停车记录和支付记录

### 管理员流程
1. 管理员登录后台
2. 查看当前车辆列表
3. 手动录入车牌入场
4. 系统从当前时刻开始计费
5. 查询历史停车记录
6. 对特殊车辆执行费用豁免

## 7. Redis 数据设计亮点

### 设计思路
- Redis 没有关系型数据库中的表结构，因此本项目采用 Key 命名规范 + 数据类型约定方式组织数据
- 使用 `String`、`Hash`、`Set`、`List` 组合完成业务建模

### 实际 Key 示例
- `admin:{adminId}`：管理员信息
- `user:{userId}`：普通用户信息
- `user:vehicle:{vehicleId}`：车辆信息
- `user:{userId}:vehicles`：用户绑定车辆集合
- `parking:record:{recordId}`：停车记录
- `parking:current:plate:{plateNumber}`：当前在场车辆记录
- `payment:{paymentId}`：支付记录
- `auth:token:{token}`：登录会话信息

### Redis 应用价值
- 读写速度快
- 适合停车场实时状态管理
- 便于演示和快速开发
- 通过清晰的 Key 设计弥补无表结构的不足

## 8. 项目实现亮点

- 实现了完整的前后端分离结构
- 登录页面采用统一入口设计
- 当账号为 `admin` 时自动进入管理员端
- 当账号不是 `admin` 时自动进入普通用户端
- 管理员可以手动录入车辆入场
- 对未绑定用户的车辆，也可以作为临时车辆入场管理
- 系统会从车辆入场的当前时刻自动开始计费
- 前端增加了输入合法性校验，提高了系统稳定性和用户体验
- Redis 中已支持演示数据自动初始化，方便项目展示

---

## 9. 安全与规范设计

- 前端实现账号、密码、手机号、车牌号、车位号等格式校验
- 后端实现统一参数校验和异常处理
- 登录后使用 Token 维持会话
- Token 会话保存在 Redis 中，并设置过期时间
- 系统已设计后续密码哈希和敏感信息加密存储方案

## 11. 项目难点分析

- Redis 没有关系型数据库表结构，需要自行设计数据组织方式
- 停车时长和费用需要根据时间动态计算
- 管理员和普通用户权限不同，需要做角色区分
- 前后端分离后需要解决接口联调与页面跳转控制问题
- 既要支持绑定车辆，也要支持临时车辆入场

---

- 项目已经打通了“登录 -> 入场 -> 计费 -> 查询 -> 缴费”的核心流程
- 管理员和普通用户两类角色的业务逻辑清晰
- 系统具备后续继续扩展和完善的基础
